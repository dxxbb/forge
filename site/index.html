Flows

数据流

当前页面最容易讲混的地方就在这里：结构是三块，流有三条，不能混着看。

写入流

新信息怎么变成稳定内容

1

进入 Capture

user input / platform note / raw import

2

Consolidate

判断是 profile / project / decision / topic / procedure，还是丢弃 / 归档

3

写入 Store

稳定、当前有效的内容在这里固定下来

4

Project

从 Store 刷新各平台 projection

使用流

平台平时实际读什么

1

平台读入口文件

Claude 读 CLAUDE.md，OpenClaw 读 MEMORY.md，Codex 读 AGENTS.md，ChatGPT 读你送进去的 summary

2

agent 开始工作

Phase 1 默认先用 projection，不假设平台会自动读本地 Store

3

需要更深内容时再显式取

本地平台可额外打开相关文件或走 MCP；ChatGPT 默认做不到这一步

回流

平台上的新结论怎么回主库

1

平台侧产生新规则

比如在 Claude Code 里确认“每节保留 1 个例子”

2

回写 Capture

先记成 correction item 或 daily note，不直接覆盖 Store

3

需要当天生效时先补 projection

只刷新目标平台入口，不等晚上统一整理

4

再 Consolidate + Project

晚些时候并进 Store，再统一刷新所有相关 projections

Platform Landing

各平台落地图

Store 不直接暴露给平台。每个平台都有自己的正式入口。

Store

唯一真相源

Claude Code

~/.claude/CLAUDE.md + repo CLAUDE.md

projection 进入项目 memory 体系，可用 hooks 做当天补充

OpenClaw

MEMORY.md + memory/YYYY-MM-DD.md

workspace 本身就是落点

Codex

AGENTS.md + config.toml + MCP

不能只放 projection 文件，必须写到它会发现的入口

ChatGPT

project files / instructions / memory

只吃你送进去的 summary，送达状态要单独记录

Build Plan

这个项目真正要开发什么

先把结构和同步做对，再做 fancy retrieval。

Phase 1

Capture / Store / Projections 结构、文件操作、模板、例子。

Phase 2

导入器、source refs、normalization、correction lifecycle。

Phase 3

Consolidate jobs、conflict handling、archive routing。

Phase 4+

platform sync、MCP、optional indexes、evolve pipeline。
